# التوثيق النهائي الشامل — Smart Space Booking Platform (Arabic)

> وثيقة تنفيذية + تقنية موجهة للعميل، الإدارة، والفِرق المتخصصة (Backend / Web / Mobile / QA / DevOps).
> 
> **الهدف:** شرح المنصة بالكامل بلغة واضحة غير برمجية بحتة، مع عمق هندسي كافٍ لإقناع أي مراجع تقني متخصص.

---

## 1) الملخص التنفيذي

منصة **Smart Space Booking** هي نظام إدارة وتشغيل وحجز مساحات ذكي (قاعات، مكاتب، مساحات عمل مشتركة، مخازن...) يقدم دورة تشغيل كاملة من أول استفسار العميل وحتى التحصيل المالي والتقارير.

### القيمة التجارية المباشرة
- رفع نسبة الإشغال وتحسين الاستفادة من المساحات.
- تسريع دورة البيع (Lead → Quotation → Booking → Payment).
- تقليل الأخطاء البشرية في الإتاحة والتسعير والعقود.
- تحسين تجربة العميل عبر رحلة واضحة وشفافة.
- إتاحة مؤشرات أداء دقيقة لدعم القرار.

---

## 2) أهداف المشروع (Objectives)

### أهداف العميل/الإدارة
1. مركزية إدارة كل الفروع والمساحات في مكان واحد.
2. تسعير ذكي قابل للتخصيص حسب الوقت/النوع/السعة/الإضافات.
3. تقليل تضارب الحجوزات وتفادي الحجز الزائد.
4. رقمنة العقود والفواتير والمدفوعات.
5. تقارير تشغيلية ومالية لحظية.

### أهداف هندسية
1. Architecture نظيفة قابلة للتوسع.
2. Modular Domain Design لتقليل التشابك.
3. Workflow engines بدل CRUD العشوائي.
4. API-first جاهزة لتطبيق ويب وموبايل.
5. تصميم يدعم التكاملات المستقبلية (دفع، إشعارات، تقاويم).

---

## 3) أصحاب المصلحة (Stakeholders)

- **Owner / Management:** مراقبة الربحية والإشغال.
- **Operations Team:** إدارة الفروع والمساحات والحجوزات.
- **Sales Team:** إدارة leads وعروض الأسعار والتحويل.
- **Finance Team:** الفواتير، التحصيل، الاسترداد.
- **Customer:** استعراض العرض، الحجز، الدفع، التوقيع.
- **Technical Teams:** Backend, Frontend, Mobile, QA, DevOps.

## 5) المعمارية العامة (Architecture)

### 5.1 نمط البناء
- Laravel modular architecture.
- تقسيم النظام لموديولات أعمال مستقلة.
- Shared Kernel للمكونات المشتركة.

### 5.2 لماذا هذه المعمارية؟
- سهولة التطوير المتوازي بين الفرق.
- تقليل تأثير التعديلات على بقية النظام.
- سهولة إضافة خصائص مستقبلية بدون إعادة هيكلة جذرية.

### 5.3 مبادئ الجودة
- Single responsibility per module/service.
- Domain logic داخل Engines/Services.
- API responses موحدة.
- Exceptions واضحة ومفهومة.
- قابلية اختبار عالية.

## 8) العلاقات الأساسية بين الكيانات (Relations)

```mermaid
erDiagram
    CUSTOMER ||--o{ QUOTATION : requests
    QUOTATION ||--o{ QUOTATION_ITEM : contains
    QUOTATION ||--|| BOOKING : converts_to
    BOOKING ||--o{ BOOKING_ITEM : contains
    BOOKING ||--|| CONTRACT : has
    BOOKING ||--|| INVOICE : billed_by
    INVOICE ||--o{ PAYMENT : paid_by
    PAYMENT ||--o{ PAYMENT_REFUND : may_have
    SPACE ||--o{ BOOKING_ITEM : reserved_in
    BRANCH ||--o{ SPACE : owns
```

### ملخص مفهومي للعلاقات
- العميل قد يملك عدة عروض أسعار وحجوزات ومدفوعات.
- العرض يمكن تحويله لحجز.
- الحجز ينتج عنه عقد وفاتورة.
- الفاتورة تستقبل دفعة واحدة أو عدة دفعات حسب السياسة.

## 9) الـ Workflows التشغيلية كاملة

## 9.1 رحلة المبيعات حتى التحصيل
```mermaid
sequenceDiagram
    participant Sales
    participant System
    participant Customer
    participant Finance

    Sales->>System: إنشاء عرض سعر
    System-->>Sales: عرض السعر + الإتاحة + التسعير
    Customer->>System: قبول العرض
    System-->>Sales: إنشاء حجز
    System-->>Customer: عقد للتوقيع
    System-->>Finance: إصدار فاتورة
    Customer->>System: دفع
    System-->>Finance: إيصال + تحديث حالة
    System-->>Sales: تحديث Funnel
```

## 9.2 إدارة الإتاحة
1. تحديد الفترة الزمنية المطلوبة.
2. فحص التداخلات الحالية.
3. احتساب السعة المستخدمة والمتبقية.
4. إرجاع نتيجة متاحة/جزئية/غير متاحة.

## 9.3 إدارة التسعير
1. سعر أساسي للمساحة.
2. معاملات حسب النوع/الوقت/السياسات.
3. خصومات حسب المدة أو الحملات.
4. إضافة خدمات اختيارية.
5. ضريبة ثم إجمالي نهائي.

## 9.4 إدارة العقود
- توليد عقد بناءً على نوع الحجز.
- جمع توقيع الأطراف.
- تحويل الحالة إلى Signed.

## 9.5 إدارة المدفوعات
- تسجيل الدفعة.
- إصدار إيصال.
- مزامنة حالة الفاتورة/الحجز.
- السماح بالاسترداد الجزئي أو الكامل وفق القيود.

## 11) الواجهة الخلفية، الويب، والموبايل

## 11.1 Backend
- مسؤول عن قواعد العمل، الأمان، والتكاملات.
- يقدم APIs موحدة وقابلة للإصدار (V1/V2).
- يضمن الاتساق عبر transactions وstatus transitions.

## 11.2 Web Application
- لوحة تشغيل للإدارة والموظفين.
- واجهات إدارة CRUD + شاشات تشغيل (Calendar, Funnel, Revenue).
- صلاحيات حسب الدور.

## 11.3 Mobile Application
- تجربة عميل سريعة: طلب، قبول، دفع، متابعة.
- تجربة ميدانية للموظف: تأكيد حضور، تحديث حالة، إشعارات فورية.
- يعتمد على نفس الـ APIs لضمان سلوك موحد.

## 18) Glossary مبسط

- **Quotation:** عرض سعر.
- **Booking:** حجز مؤكد/قيد التنفيذ.
- **Invoice:** مطالبة مالية رسمية.
- **Refund:** إرجاع مالي.
- **Occupancy:** نسبة إشغال.
- **Funnel:** رحلة التحويل البيعي.

## 19) الخلاصة للعميل

هذا النظام مصمم ليكون **منصة تشغيل أعمال متكاملة للمساحات** وليس مجرد شاشة حجز.
يجمع بين:
- وضوح تجربة العميل.
- انضباط العمليات الداخلية.
- صلابة مالية وتشغيلية.
- قابلية نمو طويلة الأجل.

**النتيجة:** منصة جاهزة للنمو، قابلة للتخصيص، وتدعم قرارات الإدارة بأرقام دقيقة وموثوقة.
